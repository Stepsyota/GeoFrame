# GeoFrame

**GeoFrame** — это Telegram-бот и backend-проект для работы с фото и видео: он помогает собирать медиаданные, хранить их в базе и готовить структуру для публикации в Telegram-темах.

> GeoFrame задуман как личный помощник и альтернативное “облачное хранилище” для фотографий из поездок и повседневной жизни. Идея проекта в том, чтобы не разбирать вручную тысячи снимков и не платить за хранение в сторонних облачных сервисах: достаточно поместить все фото в одну директорию, запустить обработку и получить готовый Telegram-чат, где каждая тема соответствует одному дню съёмки. Внутри темы фотографии автоматически сортируются по времени и дополняются полезными подписями на основе метаданных, например датой, временем или местом съёмки.

> В дальнейшем GeoFrame может развиться в умного фото-помощника с нейросетевыми функциями: распознаванием лиц, тегированием по содержимому кадра и выбором лучших снимков, чтобы не хранить десятки почти одинаковых фотографий.

## ✨ Что это за проект

Сейчас GeoFrame уже умеет:

- работать как Telegram-бот на `aiogram 3`
- хранить данные в PostgreSQL через `SQLAlchemy` и `asyncpg`
- сканировать локальную директорию с фото и видео
- извлекать метаданные файлов
- считать `sha256` и `phash` для изображений
- хранить темы, сообщения и связи между Telegram и медиа
- покрывать базовые CRUD-сценарии тестами

Идея проекта уже читается довольно ясно: это фундамент для системы, которая связывает локальный архив медиафайлов и Telegram-форум с понятной структурой данных.

## 🧩 Что уже реализовано

### Индексация медиа

Проект проходит по локальной папке с файлами и для каждого файла собирает:

- путь
- размер
- время создания в файловой системе
- время съёмки, если оно есть в метаданных
- разрешение медиафайла
- GPS-координаты, если они доступны
- `sha256` для точной идентификации
- `phash` для изображений
- длительность для видео

### Работа с Telegram-структурой

В базе уже заложены сущности для:

- Telegram topics
- Telegram messages
- связи между сообщением и медиафайлом

Это даёт проекту важную опору: он может работать не только с файлами как с набором картинок, но и с их будущим размещением в Telegram.

## 🧠 Какие данные хранит проект

### Медиа

Медиаданные в проекте построены вокруг нескольких связанных таблиц.

#### `media_files`

Это основная таблица, в которой хранится запись о каждом найденном медиафайле. По сути, это общая карточка файла, независимо от того, является он изображением или видео.

В таблице хранятся:

- `storage_path` — путь к файлу в локальном хранилище
- `size_bytes` — размер файла в байтах
- `created_at_fs` — время создания файла в **файловой системе**
- `taken_at` — время съёмки, если его удалось извлечь из метаданных
- `sha256` — уникальный хеш файла для точной идентификации и дедупликации
- `width` и `height` — размеры изображения или видеокадра
- `gps_lat`, `gps_lon`, `altitude` — координаты и высота, если они присутствуют в метаданных
- `media_type` — тип файла: `image` или `video`

Именно `media_files` является центральной таблицей медиаслоя, вокруг которой строятся дополнительные специализированные сущности.

#### `media_images`

Эта таблица дополняет `media_files` данными, которые относятся только к изображениям.

В ней хранится:

- `media_id` — ссылка на запись в `media_files`
- `phash` — perceptual hash изображения

`phash` нужен не для точного совпадения байтов, а для поиска визуально похожих изображений. Это хорошая основа для будущего отбора дублей и почти одинаковых кадров.

> Связь между `media_files` и `media_images` — `1:1`: у изображения может быть одна дополнительная запись с image-специфичными данными.

#### `media_videos`

Эта таблица хранит данные, которые относятся только к видео.

В ней находятся:

- `media_id` — ссылка на запись в `media_files`
- `duration` — длительность видео в секундах

> Связь между `media_files` и `media_videos` тоже `1:1`: если файл является видео, его дополнительная информация хранится здесь.

#### `media_live_photos`

Таблица нужна для объединения двух отдельных файлов в одну логическую сущность live photo:

- `image_media_id` — ссылка на фото-компонент
- `video_media_id` — ссылка на видео-компонент

Это позволяет не просто хранить картинку и короткий ролик отдельно, а понимать, что они представляют одну и ту же live photo-пару.

### Telegram

Telegram-часть проекта хранится отдельно от медиаданных, но связана с ними через сообщения.

#### `telegram_topics`

Таблица описывает темы Telegram-форума, в которые бот может публиковать фотографии.

В ней хранятся:

- `chat_id` — идентификатор Telegram-чата
- `topic_id` — идентификатор темы внутри чата
- `name` — название темы
- `kind` — тип темы, например `original` или `day_photo`

Эта таблица нужна, чтобы бот мог не просто отправлять файлы в чат, а раскладывать их по нужным темам осмысленно и предсказуемо.

#### `telegram_messages`

Здесь хранится информация об отдельных сообщениях, отправленных в Telegram.

В таблице есть:

- `message_id` — идентификатор сообщения в Telegram
- `topic_id` — ссылка на запись в `telegram_topics`

Таким образом проект знает, какое сообщение относится к какой теме.

> Связь между `telegram_topics` и `telegram_messages` — `1:N`: одна тема может содержать много сообщений.

#### `media_telegram_messages`

Это связующая таблица между локальными медиафайлами и отправленными Telegram-сообщениями.

В ней находятся:

- `media_id` — ссылка на запись в `media_files`
- `telegram_message_id` — ссылка на запись в `telegram_messages`

Она нужна для того, чтобы можно было отследить, какой именно файл был отправлен каким сообщением. Это особенно важно для повторной синхронизации, анализа уже загруженных медиа и дальнейшей автоматизации публикации.

Конфигурация пока минимальная и сосредоточена вокруг доступа к Telegram и подключения к основной и тестовой базе данных.

## 🚀 Быстрый старт

### Установка зависимостей

```bash
pip install -r requirements.txt
```

### Запуск бота

```bash
python -m app.main
```

## 🗄 Работа с базой данных

Для создания таблиц есть отдельный скрипт:

```bash
python -m app.db.create_tables
```

> ⚠️ Важно: в текущем виде скрипт удаляет все существующие таблицы и создаёт их заново.

## 📁 Назначение каталогов

- `app/bot` — команды бота, обработчики и сценарии взаимодействия с Telegram
- `app/db` — модели базы данных и CRUD-операции
- `app/utils` — утилиты для анализа файлов, хеширования и извлечения метаданных
- `app/services` — сервисный слой и orchestration-логика
- `tests` — автотесты для базовых сущностей и связей
